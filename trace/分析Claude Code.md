# 拆解 Claude Code：用 mitmproxy 與 Skill 一窺 AI 編程助手的內部運作

> 你有沒有好奇過，當你在終端機裡對 Claude Code 下一道指令，它背後到底做了什麼？Harness 組了什麼提示詞？Model 又回傳了什麼決策？這篇文章記錄了我用兩種方式拆解 Claude Code 的過程與心得。

## 為什麼要分析 Claude Code？

Claude Code 是 Anthropic 推出的命令列 AI 編程工具，它能直接在終端機中幫你讀程式碼、寫程式碼、執行指令，甚至自動化整個開發工作流程。對許多開發者來說，它已經是日常開發的一部分。

但身為技術人員，光是「會用」往往不夠。我想做的事情可以歸納成三個層次：

**理解** — Claude Code 的架構長什麼樣？Harness 跟 Model 之間怎麼溝通？系統提示詞裡藏了什麼設計思路？

**學習** — 它的 Tool 機制怎麼運作？Skill 系統如何載入？這些設計模式能不能應用到自己的專案裡？

**仿作** — 理解了架構之後，能不能自己搭一套類似的 Agentic 工作流程，或者針對特定場景做出更貼合需求的工具？

帶著這些問題，我開始動手拆解。

## Claude Code 的基本架構

在深入分析方法之前，先快速了解 Claude Code 的整體架構。它大致可以拆成兩個核心角色：

```
┌─────────────────────────────────────────────┐
│              使用者 User                     │
└─────────────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────┐
│              Harness                         │
│  • 接收使用者輸入                            │
│  • 組合系統提示詞                            │
│  • 執行工具調用                              │
│  • 管理狀態和歷史                            │
└─────────────────────────────────────────────┘
                      │
          (API 調用 / API calls)
                      │
                      ▼
┌─────────────────────────────────────────────┐
│              Model (LLM)                     │
│  • 理解自然語言                              │
│  • 生成回應和決策                            │
│  • 決定使用哪些工具                          │
└─────────────────────────────────────────────┘
```

**Harness** 是在你本機端執行的程式邏輯，負責把你的輸入、系統提示詞、工具定義等資訊打包成 API 請求送給模型；同時也負責接收模型的回應，執行模型決定要呼叫的工具（例如讀檔案、跑指令），再把結果回傳給模型繼續對話。

**Model** 就是遠端的 LLM，負責理解語意、做出決策、生成回應。

這個架構的關鍵在於：所有的「智慧」來自 Model，但所有的「行動力」來自 Harness。而兩者之間的 API 通訊，就是我們分析的突破口。

## 方法一：用 mitmproxy 攔截通訊

### 為什麼用 mitmproxy？

既然 Harness 和 Model 之間是透過 API 溝通，最直覺的方式就是在中間放一個代理，把所有來回的封包都攔下來看。[mitmproxy](https://mitmproxy.org/) 正是這類工具中的佼佼者。

不過直接拿 mitmproxy 去攔 Anthropic 的 API 有個隱憂——透過 proxy 存取官方 API 可能觸發風控或違反使用條款。所以我的做法是搭配本地模型，讓整個分析過程完全在自己的機器上完成。

### 實際操作

首先，用 `llama-server` 啟動一個本地模型：

```bash
local args=(
  --model "$model_path"
  --host localhost
  --port 7080
)

llama-server "${args[@]}"
```

接著，在本地模型前面掛上 mitmproxy 的反向代理模式：

```bash
mitmproxy --mode reverse:http://localhost:7080 -p 9080
```

這樣 mitmproxy 就會監聽在 port 9080，並把流量轉發到 port 7080 的本地模型。

最後，設定環境變數讓 Claude Code 的 Harness 把請求送到 mitmproxy：

```bash
export OPENAI_API_BASE_URL="http://localhost:9080/v1"
export ANTHROPIC_BASE_URL="http://localhost:9080"
export ANTHROPIC_AUTH_TOKEN="sk-1234"
export ANTHROPIC_MODEL="llama"
```

完成之後，只要正常使用 Claude Code，mitmproxy 的介面上就能清楚看到 Harness 和 Model 之間所有的請求與回應內容——包括系統提示詞的完整結構、工具定義的 JSON Schema、多輪對話的上下文管理方式，以及模型回傳的 tool_use 決策。

### 這個方法能看到什麼？

透過攔截的內容，你可以觀察到幾件很有價值的事：

- **系統提示詞的組成方式**：Harness 如何把使用者設定、專案資訊、工具描述等組合成一份完整的系統提示詞送給模型。
- **工具呼叫的完整流程**：模型如何決定要呼叫哪個工具、帶什麼參數，以及 Harness 執行完工具後如何把結果送回。
- **多輪對話的上下文管理**：每次 API 呼叫到底送了哪些歷史訊息，哪些被裁剪了。

### 限制

因為使用的是本地模型而非 Claude 本身，模型的實際回應品質和行為會有差異。這個方法主要是觀察 Harness 端的行為，而非 Model 端的推理能力。

## 方法二：用 Skill 讓 Claude 自己說明

### 直接問 Claude Code

這個方法聽起來有點「後設」——直接在 Claude Code 裡問 Claude 自己的運作機制。例如：

- 啟動時的初始化流程是什麼？
- Tool 的載入順序是怎樣的？
- 各類設定檔放在哪裡？

由於 Claude（Model 端）能看到 Harness 傳來的系統提示詞和工具定義，它對自己「被賦予了什麼能力」是有完整認知的。這代表它可以精確地告訴你每個 Tool 的定義、參數、用途，以及 Skill 的載入機制。

### 整理成 Skill 進行反覆驗證

每次在對話中問到的資訊，會隨著對話結束而消失。所以我把這些分析過程整理成一個 [Skill](https://github.com/ElliotChen/ClaudeCode/blob/main/skills/claude-code-architecture/SKILL.md)，讓 Claude Code 可以在每次新對話中載入，持續針對架構細節進行深入分析。

這個做法的好處是可以迭代式地補強——發現有疑問的地方就修改 Skill 的提示內容，下次載入時就能針對那個環節做更深入的探查。

### 這個方法的邊界

有一點需要特別注意：Model 能看到的，僅限於 Harness 傳給它的資訊。也就是說：

- **能精確分析的**：Tool 定義、Skill 內容、系統提示詞中的指令、對話上下文的結構。
- **無法直接得知的**：Harness 內部的實作邏輯（例如 JavaScript/TypeScript 的程式碼細節）、API 呼叫前的預處理步驟、本地快取機制等。

所以這個方法更像是「從 Model 的視角向外看」，看到的是 Harness 投射給它的世界，而非 Harness 的完整實作。

## 兩種方法的互補

這兩種方法各有所長，恰好形成互補：

| 面向 | mitmproxy | Skill |
|------|-----------|-------|
| 觀察角度 | 從外部攔截通訊 | 從 Model 內部向外看 |
| 看得到什麼 | API 請求/回應的原始資料 | Tool 定義、系統提示詞語意 |
| 優勢 | 完整、不遺漏的原始資料 | 能做語意層面的分析與摘要 |
| 限制 | 需搭配本地模型，資料量大 | 受限於 Harness 傳送的資訊 |
| 適合場景 | 了解通訊協議與資料結構 | 了解工具能力與設計意圖 |

如果你想完整理解 Claude Code，建議兩種方法都試試。先用 mitmproxy 看清楚資料的「骨架」，再用 Skill 讓 Claude 幫你解讀資料的「語意」。

## 我從中學到了什麼

經過這番拆解，有幾個讓我印象深刻的觀察：

**Harness 的設計比想像中精巧。** 它不是簡單地把使用者輸入丟給模型而已。系統提示詞的組合、工具定義的注入、對話歷史的裁剪，每一步都有設計考量。

**Tool 機制是核心競爭力。** Claude Code 之所以能「做事」而不只是「說話」，關鍵在於它的 Tool 系統。理解了 Tool 的定義方式和呼叫流程，等於掌握了建構類似系統的基礎知識。

**Skill 系統讓擴展變得優雅。** 透過 Skill，使用者可以在不修改原始碼的情況下，擴展 Claude Code 的能力邊界。這個設計模式值得在其他 Agentic 系統中借鑑。

## 結語

分析一個工具的內部運作，不只是為了滿足好奇心。當你理解了 Claude Code 的 Harness-Model 架構、Tool 機制和 Skill 系統之後，你就具備了自己搭建類似系統的知識基礎，也能更有效率地使用這個工具本身。

如果你也想動手試試，可以從我整理的 [Skill](https://github.com/ElliotChen/ClaudeCode/blob/main/skills/claude-code-architecture/SKILL.md) 開始，或者直接架一個 mitmproxy 來看看那些來回的封包裡藏了什麼秘密。
